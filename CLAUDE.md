# Cairn — architecture rules

Cairn is a Flutter habit-tracking app where every task completion is proven with an AI-verified photo. Users "climb" by earning metres per completion and progress through ranks.

## UI designs are canonical — do not invent UI

The `design/` folder contains the canonical UI designs as `.dc.html` files (Today/Home, camera capture & verify flow, Trail, New Habit variants, Stats, Profile, Premium, onboarding, empty states, failure states, daily limit). **All real screens must be implemented from those files** — palette, typography (Zilla Slab + Work Sans), pebble shapes, and exact copy come from them. No screen designs may be invented. The only exception is the Phase 1 debug screen, which is deliberately ugly/minimal and does not use the designs.

Full rationale for every decision below lives in `docs/proof-todo-app-guideline.md`. That document is the source of truth; this is the sequencing. Phases may be re-planned: if you change them, rewrite this section so future sessions inherit the current map. Mark phases complete as you finish them, and record any decisions made along the way.

- **Phase 1: Data foundation. ✅ COMPLETE.** drift database, occurrence generation, streaks, metres/rank, repositories, `LocalDate`/`Clock` abstractions, minimal debug screen. 55 tests green. Not yet run on a physical device.

- **Phase 2: Proof pipeline.** Split so the agent-buildable core lands before anything that needs the human.
  - **2a: Client pipeline behind a `ProofVerifier` interface (no network, fully testable). ✅ COMPLETE** (156 tests green; smoke-tested on a physical Android 14 device: camera and gallery capture, compression, caps, offline/pending/retry, and recency all behave). Phase 1 audit fixes first (partial unique index so a tombstoned completion frees its slot; input validation in the repositories). Then: camera *and* gallery capture via `image_picker` (both always available for every task; there is no per-task restriction). Compress to ~150 KB for the wire. `photo_manager` recency pre-filter on the photo library's capture time. Verification attempts recorded in their own sync-ready table (a rejection is not a completion; the UNIQUE constraint on completions makes retries impossible otherwise). Enforce **5 successful proofs/day** and **3 attempts/task/day** (rejections do NOT burn the daily cap). Offline or verifier unavailable: `verification_status = 'pending'`, retry on reconnect/foreground; a pending completion counts toward streaks and caps optimistically, and is tombstoned (attempt recorded) if the retry comes back rejected. A configurable fake verifier drives tests and the debug screen.
    - Decisions made while building 2a: all proof policy lives in `ProofPolicy` (thresholds, caps, **and** the recency window), and every gate is enforced in `CompletionRepository.completeWithProof` rather than the UI, so no caller can bypass it. Guard order: back-fill, task exists, scheduled, duplicate, attempts cap, daily cap, recency, *then* the verifier. A stale photo never reaches the verifier and never writes an attempt row: **recency failures do not burn the attempts budget**, only verifier rejections do. An **unknown** photo timestamp fails *open* (`allowUnknownPhotoTime`, default true): `photo_manager` lookups legitimately return nothing on limited-permission or cloud-only libraries, and Gemini is the real gate. Recency window: 15 min. Photo lifecycle: verified and pending completions keep the file (a pending one needs it to retry), any rejection deletes it. `ProofFlowService` (capture, compress, save, complete) is the single entry point the Phase 3 UI calls; `ProofRetryTrigger` runs the pending retry on app foreground and on regaining connectivity.
    - Learned on device: `photo_manager`'s capture-time lookup alone is **not** enough on Android 13+, where `image_picker` goes through the system Photo Picker and hands the app a renamed copy in its cache, so the library match fails and every gallery pick fell open to "unknown". The resolver is therefore a chain: photo library first (its timestamp is hardest to forge), **EXIF `DateTimeOriginal` as the fallback** (the guideline's own nominated fallback). With that in place, stale gallery picks (photos and screenshots alike) are rejected and fresh ones pass, verified on device. Caps and attempt counts are exposed as repository methods (`successfulProofsToday`, `attemptsUsedToday`) because Phase 3's Daily Limit screen needs the same numbers.
  - **2b: Real verifier. ✅ COMPLETE.** Deployed (`verify-proof`, `verify_jwt` on) and verified end to end on a physical device: an honest photo passes, an unrelated photo is rejected with Gemini's own reason and burns an attempt, a screenshot verifies on its merits, and the Edge Function logs show 200s with no parser fallback (about 4s per round trip). Anonymous auth, `user_id` stamping + one-time backfill, and `SupabaseProofVerifier` are all live. The debug screen keeps `pass`/`reject`/`offline` fake modes alongside the default `real` mode, so device testing never has to burn Gemini calls. Anonymous Supabase auth on first launch, moved up from Phase 4: the Edge Function must verify a caller JWT and anon auth is what supplies it. Assign `user_id` from the first session so the Phase 4 upgrade preserves it. A **Supabase Edge Function** holds the Gemini API key (never in the app binary), verifies the caller's anon JWT, rate-limits (best-effort in-memory until Phase 4 tables exist), and returns structured JSON: `task_shown`, `confidence`, `is_screenshot_or_screen`, `reason`. Client-side mapping: `verified` iff `task_shown && confidence >= threshold` (default threshold 0.6, configurable). **`is_screenshot_or_screen` is recorded metadata only and never rejects on its own** (decided 2026-07-14): an earlier design had Gemini judge whether a screen was the *natural* evidence for the task, but that asks the model to guess which proof methods are legitimate and would reject honest ones nobody anticipated. The cost is a known, accepted bypass (screenshot an old photo and its capture time looks fresh, so recency passes); the guideline's own position is that the goal is friction, not a perfect lock, and a determined cheater can stage a shot regardless. Model: **Gemini 3.1 Flash-Lite**, minimal thinking. Every completion is AI-verified; there is no honor-system fallback.
  ⚠️ **Requires the human:** a real-device run of the 2a capture path (camera, gallery, `photo_manager` capture-time lookup and compression are all plugin-backed, so `flutter test` cannot exercise them; the debug screen's verifier-mode switch drives pass/reject/offline without a network). For 2b: Supabase project URL + anon key wired into the app config, Gemini API key, `supabase functions deploy` + secrets.

- **Phase 3: Real UI** from `design/`. Implement every screen from the canonical `.dc.html` files. Screens are largely independent, so this phase parallelizes well across sequential implementer runs (one per screen).

- **Phase 4: Accounts + sync.** Optional email/password upgrade of the Phase 2b anonymous user that **preserves the same `user_id`** so history carries over. Postgres tables mirroring the local schema, with **row-level security** per user. **Hand-rolled delta sync**: cursor pull on `updated_at` + dirty push on foreground; honor `deleted_at` tombstones; last-write-wins. Durable server-side rate limiting and the server-side no-back-fill guard land here too. **Before any public release** (deferred here by decision, 2026-07-14): enable **captcha on anonymous sign-ins** and add a **global** ceiling to the Edge Function. The 2b limiter is per user id and in-memory, so scripted anonymous sign-ups mint unlimited fresh identities, each with a fresh budget, each calling `verify-proof`; every call is a paid Gemini request. The exposure is the Gemini bill, not just Supabase MAU. No PowerSync, no realtime. **Record sync is free**; cloud photo backup is premium and post-MVP.
  ⚠️ **Requires the human:** Supabase project config, RLS review.

### Standing constraints (all phases)

- The schema is **sync-ready from day one**: client-generated UUID v7 PKs, nullable `user_id`, `updated_at` epoch-millis for last-write-wins, `deleted_at` tombstones (never hard-delete).
- A Supabase MCP server is connected for **inspecting** the remote project. **Migrations go through the Supabase CLI** (`npx supabase migration new` / `db push`) — MCP cannot manage migration files. Do not create remote tables until Phase 4.
- Anything marked ⚠️ cannot be completed by an agent alone. Stop and hand back rather than stubbing it out.

## Stack

- Flutter; **Riverpod** for state; **drift** (SQLite) for the local DB.
- **rrule** package for weekly and monthly-nth-weekday recurrence *only*. Monthly day-of-month is hand-rolled (see below).
- `uuid` for UUID v7 IDs; `timezone` package where an explicit zone is needed (tests).

## Domain rules (do not violate)

- **Local calendar dates everywhere.** `occurrence_date`, streaks, and day boundaries use the user's *local* calendar date (day flips at local midnight). Never UTC dates for these. The `LocalDate` value type in `lib/src/models/local_date.dart` and the `Clock` abstraction (`lib/src/clock.dart`) exist for this — always get "today" from a `Clock`, never `DateTime.now()` directly in domain logic.
- **An occurrence is `(task, localDate, slot)`.** `due_times` is a JSON list of `"HH:mm"` strings; its index is the slot. Empty list = one untimed slot 0.
- **Monthly `day_of_month` clamps, never skips.** `day = min(month_day, lastDayOfMonth)`. A task on the 31st fires Feb 28 (29 in leap years). Standard RRULE `BYMONTHDAY` skips short months, which is why this generator is hand-rolled and must never be replaced with rrule.
- **Monthly `nth_weekday`** uses rrule `FREQ=MONTHLY;BYDAY=3FR` style; `month_nth = -1` means "Last" (`BYDAY=-1FR`).
- **No back-filling.** Completions may only be created for occurrences whose `occurrence_date == today` (local). Enforced in the repository layer.
- **One proof per slot per day:** `UNIQUE(task_id, occurrence_date, slot)` on completions.
- **Streaks are derived, never stored.** Computed on read by walking scheduled dates backward from today. A date counts iff *every* slot that date is complete (all-or-nothing). Today-pending doesn't break the streak; only a fully-elapsed incomplete scheduled date does. Non-scheduled dates are skipped, not misses. No midnight jobs, no stored counters.
- **Metres (`points_awarded`)** are computed at completion-insert time: base 10 m; + per-task streak bonus (+1 m per consecutive day of that task's streak counting today, so a first-ever completion earns +1 m; capped +10 m); + perfect-day bonus (+15 m attached to the day's final scheduled occurrence across all tasks). Total altitude = `SUM(points_awarded)` over non-tombstoned **verified** completions only.
- **A pending completion earns no metres until it is verified** (decided 2026-07-14). Its `points_awarded` is still computed and stored at insert time (the rule above stands) and it still counts toward **streaks**, the **daily cap** and the **perfect-day bonus**, so an offline user never loses a streak for something outside their control and nobody can farm unlimited pending completions past the 5/day cap. It simply contributes no altitude until the verdict lands, at which point the metres it was awarded at insert time start counting, unrecomputed. This is what makes the guideline's promise that **rank never decreases** true: a proof that is later rejected can no longer claw back metres, because it never counted any. `pendingAltitude()` exposes the withheld total. The UI treatment is now canonical in `design/`: Home shows an "Awaiting verification" chip and `9 stones · 13 m when verified` (the stone is placed, the metres wait), Profile shows `+13 m awaiting verification` under the total, and **`Cairn Verify Pending.dc.html`** is the post-capture outcome screen when the verifier is unreachable ("Saved. We'll verify it soon.", "Streak safe / counts today", "13 m held / lands on verify"). Note the design vocabulary: a **stone** is one completion (it builds a task's cairn), **metres** are the cumulative score and rank.
- **Per-task cairns (the "Cairn N · M stones" vocabulary), decided 2026-07-16.** A task's stones (its non-tombstoned completions, verified or pending alike) stack into numbered **cairns**, each holding up to **10 stones** (`PointsService.cairnCapStones`, tunable). Stones fill the current cairn in completion order; when it reaches 10 the cairn **caps** (a finished monument, plus the bonus below) and the next stone begins the next cairn. When the day-level **streak breaks** (a fully-elapsed scheduled date left incomplete) the current cairn is sealed early as **broken** with fewer than 10 stones, and the next completion starts a fresh one. The cairn a task is currently filling is **"Growing now"**; the very first is the **trailhead**. So **"Cairn N · M stones" means the task's Nth cairn with M stones placed in it**: N counts the cairns built so far plus the one in progress, M is the stones in that current cairn (0..9). It is NOT the task's creation order, nor its lifetime stone total. Home shows each task's current growing cairn; the Trail (`Cairn Trail.dc.html`) shows the selected task's whole ordered history of cairns (capped / broken / growing / trailhead). Cairn grouping is **derived on read** from completions plus scheduled dates, never stored (same discipline as streaks). (Follow-up: the Home Today cards still label "Cairn N" by task creation order and "M stones" by lifetime total, the pre-cairn-model reading; unifying them onto this model is a pending fix. The two readings coincide only while a task is still on its first cairn.)
- **Capping a cairn awards a bonus** (decided 2026-07-16): the stone that fills a cairn to its 10th earns an extra **+25 m** (`PointsService.cairnCapBonus`, tunable) on top of base + streak + perfect-day, computed and stored in that completion's `points_awarded` at insert time like every other bonus and bound by the same rules: withheld while pending, counted only once verified, never recomputed. Because the bonus lives on one specific stone, a rejection that tombstones that stone takes the bonus with it (the cairn falls back below 10 and no other stone inherits it). This is the same accepted "computed at insert, never recomputed" drift the streak bonus already carries, and it keeps rank monotonic.
- **Rank tiers** (cumulative metres): Pebble 0 · Cairn 150 · Ridge 450 · Crag 1,100 · Bluff 2,400 · Peak 5,000 · Summit 8,849.
- In Phase 1 completions are inserted via a debug action with `verification_status = 'verified'`.

## Layout

- `lib/src/db/` — drift tables + `AppDatabase` (codegen via `dart run build_runner build`)
- `lib/src/models/` — `LocalDate`, `Occurrence`, enums
- `lib/src/services/` — occurrence generator, streaks, points/rank
- `lib/src/repo/` — task + completion repositories (all writes go through here)
- `lib/src/debug/` — Phase 1 debug screen
- `test/` — unit tests are the acceptance criteria; run `flutter test` after every change.

## Git

- **Never run `git commit`, `git add`, `git push`, or any other state-changing git command.** The human owns all commits.
- After completing a phase or a meaningful step, **print a suggested commit message** in a code block for the human to copy. Use conventional-commit style (`feat:`, `fix:`, `test:`, `chore:`), a concise subject line, and a short body listing what changed and why.
- Read-only git commands (`git status`, `git diff`, `git log`) are fine.

## About "—"

Do not ever use the character "—". Use another fitting character.
